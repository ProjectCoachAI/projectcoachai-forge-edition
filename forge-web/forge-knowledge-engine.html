Foundation

What the Knowledge Engine is

The Forge Knowledge Engine is a structured, versioned intelligence layer that sits between user queries and AI model responses. It gives Forge domain knowledge that no individual AI model possesses — knowledge about Forge itself, about how to interpret multi-AI consensus, and about the specific domains users work in. As Forge is used, this layer grows. As it grows, every user benefits.

⚠ Human review gate is mandatory. Forge never ingests its own outputs automatically. You are the editor. Forge is the writer. The knowledge base is the publication.
Progression Model

Four levels of intelligence

The Knowledge Engine evolves through four levels. Each level uses and builds upon the one before it. Levels 1 and 2 are built this week. Levels 3 and 4 follow post-launch as usage data accumulates.

01
Static Knowledge
Build this week

Structured knowledge modules written and curated by humans, stored as versioned JSON/Markdown files. Injected as system context into every relevant Forge query. Forge knows what you teach it.

"We write it. Forge uses it. Users benefit from it immediately."
Five seed modules created this week (AI technology, decision frameworks, data quality, AI limitations, Swiss business context)
Each module has a unique ID, version, confidence rating, domain tag, and review date
Stored as flat files in the Forge codebase — no new infrastructure required
Injected into synthesis prompt as system context when domain matches query type
02
Synthesis-Derived Knowledge
Build this week

Forge's own Best-of-Best synthesis outputs are reviewed by a human, approved, structured into the module schema, and ingested back into the knowledge base. Forge learns from its own best answers. This is what was demonstrated with the chatbot technology synthesis.

"Forge generates. Human approves. Forge ingests. The loop closes."
Every high-quality synthesis output is a candidate knowledge module
Human review gate: approved outputs are structured and ingested, rejected ones are discarded
Each ingested module is tagged with source: "forge-synthesis" for traceability
Modules are versioned — updated when Forge produces a better synthesis on the same topic
03
Usage-Derived Knowledge
Post-launch

Patterns from real user questions across thousands of sessions reveal what knowledge gaps exist. The most-asked question types that produce low consensus or poor synthesis quality flag where new modules are needed. Forge learns from what users actually need — not just what we anticipate.

"Usage data reveals gaps. Gaps trigger module creation. Modules close the gaps."
Query clustering identifies common question categories
Low consensus scores on specific query types flag knowledge gaps
High-frequency question types without existing modules trigger new module creation
Module quality is measured by whether synthesis quality improves after ingestion
04
Autonomous Improvement
Roadmap

Forge identifies gaps in its own knowledge base, generates candidate knowledge modules, and surfaces them for human review and approval. The human remains the editor at all times — but Forge now actively participates in its own education. This is where Forge becomes a genuinely intelligent engine, not just a pass-through.

"Forge identifies what it doesn't know. Proposes solutions. Humans approve. The engine self-improves."
Forge monitors its own synthesis quality scores over time
When quality drops on a topic, Forge generates a candidate module and flags for review
Human approval gate remains mandatory — no autonomous ingestion ever
Confidence scores and review dates ensure knowledge stays current and accurate

Knowledge Module Schema

Every knowledge module — whether manually written (Level 1) or synthesis-derived (Level 2) — follows this schema exactly. The schema is what transforms documents into a queryable, versionable, machine-readable knowledge system.

Module Schema JSON 
{
  // ── IDENTITY ──────────────────────────────────────────────
  "id": "FKM-001",                      // Unique ID: FKM-{number}
  "name": "AI Chatbot Technology",        // Human-readable name
  "version": "1.0",                      // Increment on every update
  "domain": "ai-technology",             // Domain tag for routing

  // ── PROVENANCE ────────────────────────────────────────────
  "source": "forge-synthesis",           // "manual" | "forge-synthesis" | "hybrid"
  "created": "2026-05-18",              // ISO date of creation
  "last_updated": "2026-05-18",         // ISO date of last update
  "review_due": "2026-08-18",           // Quarterly review by default
  "approved_by": "Daniel Jones",        // Human reviewer who approved

  // ── QUALITY ───────────────────────────────────────────────
  "confidence": "high",                 // "low" | "medium" | "high"
  "forge_consensus": 91,               // % consensus when synthesised (if applicable)
  "status": "active",                   // "active" | "draft" | "deprecated"

  // ── ROUTING ───────────────────────────────────────────────
  "inject_on_domains": [              // When to inject into system prompt
    "ai-technology",
    "model-comparison",
    "chatbot-capabilities"
  ],
  "inject_on_keywords": [            // Keywords that trigger injection
    "chatgpt", "claude", "gemini",
    "llm", "which ai", "best ai"
  ],

  // ── CONTENT ───────────────────────────────────────────────
  "summary": "One-sentence bottom line",
  "content_file": "knowledge/FKM-001-ai-technology.md",

  // ── SYSTEM PROMPT INJECTION ───────────────────────────────
  "system_prompt_snippet": "Condensed version injected into synthesis prompt",

  // ── RELATIONS ─────────────────────────────────────────────
  "related_modules": ["FKM-004"],     // Links to FKM-004 AI Limitations
  "supersedes": null                  // ID of module this replaces, if any
}

FKM-002 · Decision Intelligence
Decision Frameworks
"What are the most effective frameworks for making high-stakes decisions under uncertainty with incomplete information?"
High confidence v1.0 — ask Forge
Bottom Line

No single decision framework works for all situations. The right framework depends on three variables: how much information is available, how reversible the decision is, and how much time is available. Forge's role is to surface perspectives that inform the choice — the decision remains human.

Forge Decision Guidance
High consensus on decision
Confidence is high but not absolute — verify the shared assumption all 8 AIs may be making
Low consensus on decision
The divergence is the signal — read the outlier carefully. It may contain the most important consideration.
Irreversible decision
Apply pre-mortem and inversion before acting. Use Devil's Advocate synthesis mode.
Time pressure
Use Sweep for fastest best answer. Prioritise speed over comprehensiveness.
FKM-003 · Data Intelligence
Data & Spreadsheet Quality
"What are the most common errors in spreadsheet and CSV data and how should each type be interpreted and corrected?"
High confidence v1.0 — ask Forge
Bottom Line

Most spreadsheet errors fall into six categories. The most dangerous are silent errors — data that looks correct but isn't. Forge Excel catches these by combining AI pattern recognition with domain context from the seven context questions. Without context, AI treats anomalies as outliers. With context, it treats them as errors.

Error Taxonomy
Error typeExampleDanger levelForge response
Magnitude error42,000,000 hectares instead of 42🔴 CriticalFlag immediately — investigate coordinate or unit swap
Duplicate recordsSame row entered twice🟠 HighFlag count and location — verify before removing
Date format inconsistencyMix of DD/MM/YYYY and MM/DD/YYYY🟠 HighFlag ambiguous dates — request clarification
Missing valuesEmpty cells in required columns🟡 MediumFlag percentage missing — assess impact on analysis
OutliersValue 3 standard deviations from mean🟡 MediumFlag for review — may be valid or error depending on domain
Cascade errorsFormula errors propagating through columns🔴 CriticalIdentify root cell — all downstream values suspect
Forge Decision Guidance
Magnitude anomaly detected
Never delete without investigation. The anomaly may be a coordinate swap, unit error, or data entry mistake — each requires different correction.
High duplicate rate
Confirm deduplication rule with the user before removing — duplicates may be intentional (e.g. multiple transactions per account).
Missing values above 10%
Flag impact on analysis reliability. Recommend imputation strategy or scope limitation.
FKM-004 · AI Reliability
AI Limitations & Failure Modes
"What are the most common ways AI models fail and how should users verify, interpret, and compensate for these failure modes?"
High confidence v1.0 — ask Forge
Bottom Line

All AI models — including the eight in Forge — can fail in predictable ways. Understanding these failure modes is the difference between using AI as a tool and being misled by it. Forge's multi-model consensus approach reduces but does not eliminate these risks. The human remains responsible for the decision.

Failure Mode Taxonomy
Failure modeWhat it looks likeForge mitigation
HallucinationConfident, plausible, factually wrong outputCross-model consensus — 8 models rarely hallucinate the same wrong answer
Knowledge cutoffOutdated information presented as currentWeight real-time models (Gemini, Grok, Perplexity) for time-sensitive queries
SycophancyModel agrees with user premise even when wrongDevil's Advocate synthesis mode specifically counteracts this
OverconfidenceUncertain answer delivered with false certaintyConsensus score makes divergence visible — low consensus = genuine uncertainty
Training biasSystematic skew from training dataEight different training sets — biases rarely align across all models
Context lossForgetting earlier parts of a long conversationForge's context questions preserve domain knowledge across the session
Verification Protocol
1
Check the consensus score
Below 70% means genuine disagreement — read all responses before acting. Above 90% is strong signal but not proof.
2
Read the outlier
The model that disagrees with the consensus may have identified a genuine risk the majority missed.
3
Verify time-sensitive claims independently
For anything that could have changed in the last 6 months, use a real-time source. Gemini and Perplexity responses in Forge include live data — prioritise these for current events.
4
Apply domain expertise
Forge's synthesis is a starting point, not the endpoint. Your domain knowledge is the final filter before any high-stakes decision.
FKM-005 · Regulatory Context
Swiss Business & SaaS Context
"What are the key legal, regulatory, and operational considerations for a SaaS company incorporated in Switzerland serving international customers with AI-generated outputs?"
High confidence v1.0 — ask Forge
Bottom Line

Switzerland offers a strong legal environment for SaaS — stable jurisdiction, FADP data protection law (aligned with GDPR), VAT compliance requirements, and a credibility advantage in European enterprise markets. The key obligations for Forge are data localisation transparency, AI output disclaimers, and clear Terms of Service defining the user's responsibility to verify AI outputs before acting on them.
